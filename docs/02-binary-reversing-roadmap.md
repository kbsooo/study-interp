# 02. Binary Reversing Roadmap

## 목표

“CTF 몇 문제 풀었다”가 아니라, binary를 보고 **입력 검증 로직, control flow, data transformation을 복원하는 능력**을 만든다.

처음에는 pwn/exploit이 아니라 reversing에 집중한다.

## 환경

macOS에서도 가능하지만, reversing 자료는 Linux ELF 기준이 많다. 추천은 둘 다 갖추는 것.

### macOS 기본 도구

```bash
xcode-select --install
file ./binary
strings ./binary
otool -tV ./binary
lldb ./binary
```

### Linux/Ubuntu VM 또는 Docker에서 쓸 도구

```bash
file ./binary
strings ./binary
objdump -d -M intel ./binary
readelf -a ./binary
checksec ./binary
gdb ./binary
```

### GUI 도구

- Ghidra: 무료, decompiler 강력함
- Binary Ninja / IDA Free: 선택
- Cutter/radare2: 선택

## Phase 0 — C to Assembly 감각 만들기

직접 작은 C 코드를 만들고 컴파일한다.

```c
#include <stdio.h>
#include <string.h>

int main(int argc, char **argv) {
    if (argc != 2) return 1;
    if (strcmp(argv[1], "secret") == 0) {
        puts("correct");
    } else {
        puts("wrong");
    }
}
```

컴파일:

```bash
clang -O0 -g check.c -o check_O0
clang -O2 check.c -o check_O2
```

해야 할 것:

- `file check_O0`
- `strings check_O0`
- `otool -tV check_O0` 또는 Linux에서 `objdump -d -M intel check_O0`
- `lldb ./check_O0`

완료 기준:

- [ ] `main` 함수 시작/끝을 찾을 수 있다.
- [ ] `strcmp` 호출 위치를 찾을 수 있다.
- [ ] branch 조건을 찾을 수 있다.
- [ ] O0와 O2의 assembly 차이를 설명할 수 있다.

## Phase 1 — Static analysis 기본

### 봐야 하는 것

- 파일 포맷: ELF/Mach-O/PE
- symbol 존재 여부
- imported functions
- string literals
- main/check 함수
- branch와 loop
- table lookup / XOR / arithmetic transform

### 기본 순서

1. `file`: 무엇인지 확인
2. `strings`: 문자열 단서 찾기
3. `nm`/`readelf`/`otool`: symbol/import 확인
4. Ghidra: decompile해서 pseudo-code 보기
5. pseudo-code를 손으로 정리

### Static writeup 템플릿

```md
## Static observations

- Format:
- Architecture:
- Interesting strings:
- Imports:
- Candidate functions:

## Recovered logic

```c
// pseudo-code
```

## Hypothesis

- correct input is generated by ...
```

완료 기준:

- [ ] crackme 3개를 동적 디버깅 없이 최대한 정적으로 분석
- [ ] pseudo-code를 직접 작성
- [ ] Ghidra output을 그대로 붙이지 않고 사람이 읽게 정리

## Phase 2 — Dynamic analysis 기본

### lldb/gdb에서 할 것

- breakpoint 걸기
- run arguments 넘기기
- register 보기
- memory 보기
- instruction step
- function call 전후 값 보기
- branch 직전 flag/compare 값 보기

lldb 예시:

```lldb
(lldb) target create ./crackme
(lldb) breakpoint set --name main
(lldb) run testinput
(lldb) disassemble --frame
(lldb) register read
(lldb) memory read --format x --size 1 --count 64 $rsp
(lldb) thread step-inst
(lldb) continue
```

완료 기준:

- [ ] compare 직전 register 값을 읽을 수 있다.
- [ ] 입력 문자열이 memory 어디에 있는지 찾을 수 있다.
- [ ] branch를 우회하지 않고 정답을 복원할 수 있다.
- [ ] 필요하면 branch patch/flag 조작도 할 수 있다.

## Phase 3 — Crackme 패턴 10개

각 패턴마다 최소 1개씩 푼다.

1. plain string compare
2. reversed string
3. XOR with constant
4. per-character arithmetic
5. lookup table
6. checksum/hash-like validation
7. nested conditions
8. anti-debugging 기초
9. packed/obfuscated binary 맛보기
10. self-modifying 또는 VM-style 아주 쉬운 문제

각 writeup은 `reversing/writeups/NNN-title.md`에 저장한다.

## Phase 4 — Python으로 solver 쓰기

reversing은 손으로만 풀면 느리다. 로직을 복원하면 Python solver로 검증한다.

예시:

```python
target = [0x12, 0x34, 0x56]
ans = ''.join(chr(x ^ 0x37) for x in target)
print(ans)
```

완료 기준:

- [ ] solver script가 정답을 출력한다.
- [ ] binary에 정답을 넣으면 correct가 나온다.
- [ ] writeup에 pseudo-code와 solver를 둘 다 넣는다.

## Phase 5 — ML/AI와 연결되는 지점

초반에는 억지로 연결하지 말고, 기초가 생기면 아래 방향을 본다.

- LLM으로 decompiled code 요약/검증하기
- binary function similarity / embeddings
- malware behavior summarization
- neural decompiler 관련 논문 읽기
- “activation patching vs dynamic tracing” 비교 글 쓰기

## 흔한 함정

- Ghidra decompile 결과를 진실로 믿음
- assembly를 한 줄씩 다 읽으려 함. 먼저 control/data flow를 잡아야 함
- 정답만 찾고 writeup을 안 씀
- anti-debugging/packing에 너무 빨리 들어감
- pwn/exploit과 reversing을 섞어서 초반 목표가 흐려짐
